## Performance Overview

Cloudflare Workers run on [V8 isolates](https://developers.cloudflare.com/workers/) distributed across 330+ cities. But raw compute speed isn't everything — network latency between the user and the edge, plus cold start time, dominate the real-world experience.

### Key Findings

After benchmarking our worker endpoints across 15 global regions:

| Metric | p50 | p95 | p99 |
| --- | --- | --- | --- |
| DNS resolution | 2ms | 28ms | 85ms |
| TLS handshake | 18ms | 64ms | 142ms |
| Worker execution | &lt;1ms | 3ms | 12ms |
| Total (cold) | 89ms | 210ms | 480ms |
| Total (warm) | 21ms | 52ms | 95ms |

### Warm Start vs Cold Start

The single biggest optimization lever is **avoiding cold starts**. A warm isolate executes in under 1ms; a cold start adds 5–15ms of initialization.

**Strategies we use:**

1. **Schedule warming** — Use [Cron Triggers](https://developers.cloudflare.com/workers/configuration/cron-triggers/) to keep active isolates warm
2. **Reduce module size** — Every extra KB of JavaScript increases cold start proportionally
3. **Global variable caching** — Reuse D1 connections, fetch results, and compute across requests

### Connection Reuse with D1

```sql
-- We cache prepared statements globally
const stmt = env.DB.prepare("SELECT * FROM posts WHERE slug = ?");

async function getPost(slug) {
  const cached = cache.get(slug);
  if (cached) return cached;
  const result = await stmt.bind(slug).first();
  cache.set(slug, result, 60_000);
  return result;
}
```

### Edge Caching Strategy

We use a hybrid approach:

- **Cache API** for immutable responses (30s–5m TTL)
- **KV** for configuration data (global, eventually consistent)
- **D1** for transactional data (waitlist signups, user profiles)
- **Static assets** served directly by Cloudflare Pages

### Takeaways

1. Warm your workers — Cron triggers are free
2. Keep bundles under 100KB for minimal cold start impact
3. Use the Cache API before hitting D1 or KV
4. Profile with `wrangler tail` in production
